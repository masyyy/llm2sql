# LLM2SQL

Can a small open-source model be fine-tuned to match a frontier model at generating SQL from natural language?

This project benchmarks LLMs on text-to-SQL over a synthetic B2B SaaS database and explores GRPO and SFT fine-tuning to close the gap.

## Setup

Three conditions, identical system prompt (schema + business context definitions):

| Condition | Model | Notes |
|-----------|-------|-------|
| **Baseline** | GPT-5.2 | Frontier model, zero-shot |
| **Small base** | Qwen3-4B instruct | Zero-shot, no training |
| **Trained** | Qwen3-4B + GRPO / SFT | Fine-tuned on the benchmark schema |

**Goal:** Trained model beats base Qwen3-4B meaningfully and gets within striking distance of GPT-5.2.

## Schema

The benchmark database models a B2B sales / installed-base domain with the following tables:

- `account_managers` -- sales reps by region
- `customers` -- accounts with segment and status
- `products` -- equipment, spare parts, and service packages
- `machines` -- installed base per customer
- `service_contracts` -- contracts tied to machines
- `orders` / `order_lines` -- completed and pending orders
- `quotes` / `quote_lines` -- sales pipeline

The schema includes business context definitions (e.g. at-risk account, service penetration, churn signal) that the model must interpret correctly.

## Evaluation

20 eval queries spanning easy, medium, and hard difficulty. Each query has ground-truth SQL. Scoring:

- **Execution accuracy** -- does the generated SQL run without error? (0 or 1)
- **Result match** -- does it return the same rows/values as ground truth? (0, 0.5, or 1)
- **Score** = mean result match across all 20 queries

Evaluation compares query results, not SQL text -- column mapping and partial matching are supported.

## Project Structure

```
run_benchmark.py        # CLI: setup-db, run --model <name>
dashboard.py            # Streamlit results dashboard
modal_app.py            # Modal: serve base Qwen3-4B via vLLM
modal_train.py          # Modal: GRPO and SFT training + serving
llm2sql/
  config.py             # Env vars, model names, paths
  schema.py             # DDL and business context for the benchmark DB
  seed.py               # Create and populate benchmark.db
  prompt.py             # Build system prompt from schema
  queries.py            # 20 eval queries with ground-truth SQL
  training_queries.py   # 30 training queries (no overlap with eval)
  inference.py          # Backends: gpt-5.2, qwen3-4b, qwen3-4b-grpo, qwen3-4b-sft
  evaluate.py           # Execute SQL, compare results, score
  models.py             # Pydantic models for results
data/
  benchmark.db          # SQLite database (generated by setup-db)
  results_*.json        # Benchmark output per model
```

## Quickstart

Requires Python 3.12+ and [uv](https://docs.astral.sh/uv/).

```bash
# Install dependencies
uv sync

# Create the benchmark SQLite database
uv run python run_benchmark.py setup-db

# Run a benchmark
uv run python run_benchmark.py run --model gpt-5.2

# View results dashboard
uv run streamlit run dashboard.py
```

## Training (Modal)

Fine-tuning runs on [Modal](https://modal.com/) using vLLM for serving.

```bash
# SFT training
uv run modal run modal_train.py::sft_train

# GRPO training
uv run modal run modal_train.py::train

# Deploy fine-tuned model for serving
uv run modal deploy modal_train.py
```

## Environment Variables

Set these in a `.env` file:

| Variable | Description |
|----------|-------------|
| `OPENAI_API_KEY` | OpenAI API key for GPT-5.2 baseline |
| `MODAL_QWEN_URL` | vLLM endpoint for base Qwen3-4B |
| `MODAL_GRPO_URL` | Endpoint for GRPO fine-tuned model |
| `MODAL_SFT_URL` | Endpoint for SFT fine-tuned model |

## Key Design Decisions

- All inference uses OpenAI-compatible APIs (direct for GPT, vLLM on Modal for Qwen)
- Qwen3 thinking mode (`<think>` tags) is disabled for both training and inference
- Training queries (30) have no overlap with eval queries (20)
- Modal Volumes: `huggingface-cache` for base weights, `trained-models` for checkpoints
